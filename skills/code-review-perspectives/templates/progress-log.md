# 進捗表示ルール（共通）

実行中はユーザーがハングアップを疑わないよう、**主要ステップごとに必ず1行の進捗ログを出す**こと。観点・分類・依存追跡が多く長時間化しやすいので、特に厳密に守る。

## 絵文字の意味

- 🔍 進行中（これから/実行中）
- ✅ 完了
- ⚠️ 警告（要注意の状況、ユーザー判断を促す事象）
- ❌ エラー

## 進捗ログの書式

- **必ず 1 行**で、絵文字 + 動詞を含む短文。
- 進捗ログは**通常の出力と区別できる行**にする（前後に空行を入れるか、コードブロック外に書く）。
- 数値が把握できているなら入れる（「N 件」「M ファイル」「X 観点中の Y 個目」「深さ N」など）。
- 進捗ログ自体に分析結果を入れない（結果は所定のフォーマットで別途出力する）。

## 出すタイミング（共通の最小要件）

どのコマンドでも、最低限以下では必ず進捗を出す:

1. **各 Phase の開始時と完了時**（例: `🔍 Phase 0: 準備を開始` → `✅ Phase 0 完了（…）`）。
2. **各観点の開始時と完了時**（例: `🔍 [観点名] を評価中...` → `✅ [観点名] 完了（指摘 N 件）`）。review-repo では加えて**各分類の開始・完了**も必ず出す。
3. **時間のかかるツール実行の前**（`git log` の大量履歴取得、`rg` のリポジトリ全体スキャン、`cloc` 実行、import の大規模解析など）: `🔍 <何をしているか> 実行中...`。

進捗ログが多くノイズになりそうなときはまとめてもよいが、**少なくとも上記 1・2 は省略しない**（黙ったまま長時間経過するのを防ぐ）。

## コマンド固有の進捗ポイント

各コマンドの Phase 構成に応じた具体的な進捗文言は、各 Slash Command 側で定義し、本書式に従う。代表例:

- **review-branch**: ベースブランチ判定 / マージベース取得 / 差分取得（変更 N ファイル）/ 公開シンボルの呼び出し元検索。
- **review-repo**: ファイル数・言語構成の集計 / プロジェクト指紋取得 / ファイル分類確定（N ファイル）/ 段階1の各分類スキャン / `✅ 段階1 完了（深掘り対象の指示を待機中）`。
- **review-slice**: 起点ファイル解析 / 依存追跡（深さ N・累計 M ファイル）/ レイヤー判定 / 境界貫通検出 / `⚠️ 動的解決の可能性あり（DI コンテナ、リフレクション）` / 追跡が想定外に大きい場合（100 ファイル超等）は `--depth` での絞り込みを提案。
