# エスカレーションレポート（人間への提示）

supply-chain-attack をはじめとする「意図的な悪意の混入」を疑う観点では、検出した疑いを **1件1件レポート化**する。複数の疑いがある場合は**確信度の高い順**にまとめる。**1件もなくても、観点としてスキャンしたことを必ず明示**する（例: 「supply-chain-attack: スキャン完了、疑わしい変更は検出されず」）。

## 基本テンプレート（review-branch / review-repo 文脈）

```
### 🚨 疑い: [簡潔なタイトル]

**ファイル**: path/to/file.py:42
**追加コミット**: <SHA> by <author> (<date>)
**確信度**: 高 / 中 / 低
**カテゴリ**: ハードコードバックドア / 認証バイパス / 動的コード実行 / リバースシェル風 / 情報漏洩経路 / 隠れた転送 / 不思議な処理連鎖 / 隣接コミットギャップ / コミット署名異常 / 依存追加 / postinstall / その他

#### 観測された事実
- 具体的なコード片や挙動を引用する
- 検出のトリガーになったパターンを明示する

#### なぜ怪しいのか
- 通常の実装との違い
- 攻撃成立の経路（具体的に）

#### 代替解釈（誤検知の可能性）
- 同じパターンが正当である場合の説明
- 何を確認すれば誤検知と判断できるか

#### 推奨アクション
- マージ阻止 / 追加レビュー / 関係者確認 / ペアレビュー / SAST 実行 / 依存パッケージの監査
- エスカレーション先（セキュリティチーム、メンテナ等）
```

## スライス文脈バリアント（review-slice）

スライス評価では入口→出口の情報フローを追えるため、フロー上の位置を明示する（`slice-flow-template.md` 参照）。

```
### 🚨 疑い: [簡潔なタイトル]

**スライス内パス**: src/path/file.py:42（深さ 2、レイヤー: application）
**情報フロー**: presentation → application → ❗ ここで外部送信 → infrastructure
**確信度**: 高 / 中 / 低

#### 観測された事実
（具体的なコード片）

#### なぜ怪しいのか
（通常の実装との違い、スライス内での違和感）

#### 代替解釈（誤検知の可能性）

#### 推奨アクション
- マージ阻止 / 追加レビュー / ペアレビュー / SAST 実行 / 依存パッケージ監査
```

## 動作上の補足

- **誤検知を恐れて沈黙しない**: 疑いは疑いとして報告し、判断は人間に委ねる。「これは多分問題ない」で省略しない。
- **過剰判定も避ける**: あらゆる `eval` を Critical にしない。**コンテキスト**（外部入力の有無、認証の有無、サンドボックスの有無）を見て判定する。
- **「疑い」と「断定」を区別**: 確信できない場合は「疑い」と明示し、断定的な指摘は確実な証拠がある場合のみ。
- **専用ツールの並走を推奨**: 出力末尾に「Semgrep、CodeQL、gitleaks、Snyk 等の専用ツールとの併用を推奨」と明記する。
- **コミット署名の検証は補助的**: 署名がないこと自体は問題ではない（多くのプロジェクトで署名は任意）。**普段は署名されているのに今回だけ署名なし**といった**変化**が疑わしい。
