# 条件分岐分析テンプレート

差分を通読して「なんとなく正しそう」と判断する gestalt 読みは、境界値のズレや条件網羅漏れを見逃す。以下の4ステップで条件式を1つずつ機械的に照合する。

## ステップ1: 条件式の全列挙

対象範囲の if / switch・match / 三項演算子 / ガード節をすべて列挙する。

- review-branch: 差分中で**変更・追加**された条件式
- review-slice: スライスファイル群に**含まれる**条件式

| # | ファイル:行 | 種別 | 条件式 | 変更種別 |
|---|---|---|---|---|
| 1 | src/order.py:42 | if | `quantity >= min_order` | 変更 |

変更種別は「新規」「変更」のいずれか。review-slice では省略可。

## ステップ2: 境界値表・デシジョンテーブルの作成

各条件について on-point（境界値そのもの）と off-point（境界の直前/直後）を列挙する。

| # | 条件式 | on-point | off-point | 実装の挙動 | 期待される挙動 | 一致 |
|---|---|---|---|---|---|---|
| 1 | `quantity >= min_order` | quantity = min_order | quantity = min_order - 1 | 通過 / 拒否 | 通過 | ✅ |

複数条件が組み合わさる場合はデシジョンテーブルを作成し、各条件の真偽の組合せごとに期待結果と実装の挙動を突き合わせる。

| 条件A | 条件B | 期待結果 | 実装の挙動 | 一致 |
|---|---|---|---|---|
| true | true | 承認 | 承認 | ✅ |
| true | false | 却下 | 承認 | ❌ |
| false | true | 却下 | 却下 | ✅ |
| false | false | 却下 | 却下 | ✅ |

到達不能な組合せ（前提条件により発生し得ない行）や、複数の分岐が同じ挙動になる重複分岐もここで洗い出し、備考として明記する。

## ステップ3: 意図との照合

PR 説明・コミットメッセージ・関連仕様（docstring・README・参照 issue）から読み取れる意図を条件ごとに書き出し、実装と突き合わせる。

| # | 条件式 | 意図（出典） | 実装 | 一致 |
|---|---|---|---|---|
| 1 | `quantity >= min_order` | 「最小発注数以上で受注可」（PR説明） | `>=` で境界含む | ✅ |

出典が複数ある場合は併記する。出典間で意図が食い違う場合はその旨を報告する。

## ステップ4: 意図不明の報告

意図を裏付ける出典が見つからず照合できない条件は、推測で「問題なし」と判定せず、「意図不明」として明示的に報告する。

報告例:

```
### 意図不明: src/order.py:42 `quantity >= min_order`
- 境界値 `quantity == min_order` の扱い（許可/拒否）を裏付ける PR 説明・仕様が見つからない。
- 実装は `>=` で境界を許可側に含めているが、意図した挙動か確認が必要。
```
