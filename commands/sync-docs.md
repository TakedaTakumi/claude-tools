# Sync Docs

ブランチでの変更内容を分析し、影響を受けるドキュメントを特定して更新する。

## 目的

コードの変更に対してドキュメント(仕様書・README・ADR 等)の更新が漏れることを防ぎ、コードとドキュメントの整合性を保つ。

## 引数仕様(`$ARGUMENTS`)

- `--base=<branch>`: 差分のベースブランチ。省略時は `git rev-parse --abbrev-ref origin/HEAD` またはローカルの main / master(`git rev-parse --verify` で存在確認)を自動判定する
- 対象パス(0個以上): ドキュメント探索の範囲をこのパス配下に限定する。省略時はリポジトリ全体から自動探索する

## 実行手順

### Phase 1: 変更内容の収集

- `git rev-parse --is-inside-work-tree` でリポジトリ内であることを確認する
- ベースブランチとの差分(`git diff <base>...HEAD`)に加え、未コミットの変更(`git status --short` で一覧化し、`git diff` / `git diff --staged` で内容確認)もすべて収集する
- 変更の意図を把握するため、`git log <base>...HEAD` でブランチ上のコミットメッセージも確認する

### Phase 2: ドキュメントの探索

対象パスが指定されていればその配下、なければリポジトリ全体から更新対象候補を探索する。例:

- README、docs/ 配下の仕様書・設計書
- ADR(Architecture Decision Records)
- OpenAPI・スキーマ定義などの仕様ファイル
- CHANGELOG、運用手順書
- コードコメントから参照されているドキュメント
- 現在のブランチに紐づく PR のタイトル・本文(`gh pr view` で特定する。PR が存在しない、または `gh` が使えない場合はスキップし、その旨を報告する)

Phase 1 の変更と無関係なドキュメントは対象外とする。

候補が1件も見つからない場合は、その旨と探索した範囲を報告して終了する。

### Phase 3: 影響分析と一覧提示

変更内容とドキュメントを突き合わせ、更新が必要な箇所の一覧を提示する(表の形式は「出力形式」を参照)。PR も他のドキュメントと同じ表の1行として提示する(「ドキュメント」列に `PR #<番号>`、「該当箇所」列に「タイトル」「本文」を記載する)。

- 更新候補に挙がったが不要と判断したドキュメントも、理由とともに簡潔に報告する
- ドキュメント側の記述が正でコード側が誤っている可能性がある場合は、更新せずその旨を明示して判断を仰ぐ

### Phase 4: 確認と更新

- 一覧を提示したら、どの項目を更新するかユーザーの承認を得る
- 承認された項目のみ更新し、完了後に変更ファイルと内容の要約を報告する
- PR の更新が承認された場合は `gh pr edit <番号> --title/--body` で行う

## 出力形式

```
| ドキュメント | 該当箇所 | 現状とのずれ | 更新方針 | 優先度 |
| --- | --- | --- | --- | --- |
```

## 動作上の注意

- ドキュメントの新規作成は提案のみ行い、承認なしに作成しない
- 一覧提示後、ユーザーの承認前に編集を開始しない
- PR の更新も他のドキュメントと同様、一覧提示とユーザー承認を経てから行う
- PR が存在しない場合は PR 更新の検討をスキップする(エラーにしない)
